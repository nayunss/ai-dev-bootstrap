# 코드 품질 표준

상태: 설계 승인

## 목적

이 표준은 특정 언어의 문법 취향이 아니라 변경 비용을 낮추고 의도를 빠르게 이해하게 만드는
설계 원칙이다. 사람과 AI가 작성한 모든 코드에 동일하게 적용한다.

우선순위는 다음과 같다.

1. 정확성과 안전성
2. 읽기 쉬운 의도
3. 관심사 분리와 변경 용이성
4. 필요한 만큼의 최소 구현
5. 일관된 언어별 표현

짧은 코드 자체가 목표는 아니다. 검증, 오류 처리, 보안과 접근성을 제거해 줄 수를 줄이지 않는다.

## 언어 중립 원칙

### 읽는 사람을 위한 코드

- 이름은 구현 방식보다 도메인의 의도와 역할을 나타낸다.
- 함수와 모듈은 한 수준의 추상화에서 읽히게 한다.
- 정상 흐름을 먼저 보이게 하고 예외 흐름은 명확하게 분리한다.
- 암묵적인 상태 변경, 숨은 부작용과 호출 순서 의존을 줄인다.
- 영리한 한 줄보다 디버깅하기 쉬운 명시적 코드를 선택한다.
- 주석으로 복잡한 코드를 설명하기 전에 코드 구조와 이름을 개선한다.

### 관심사 분리

- UI 표현, 도메인 규칙, 데이터 접근과 외부 통신을 한 함수나 컴포넌트에 섞지 않는다.
- 검증, 변환, 저장, 로깅과 응답 생성을 하나의 거대한 흐름으로 묶지 않는다.
- 모듈은 하나의 명확한 책임과 변경 이유를 갖는다.
- 경계에서는 외부 데이터 형식을 내부 도메인 모델로 명시적으로 변환한다.
- 추상화는 실제로 반복되는 변경 축이나 도메인 개념이 있을 때 만든다.

관심사 분리는 파일 수를 늘리는 것과 같지 않다. 함께 변경되고 함께 이해되는 작은 로직은
가까이 둔다. 단순한 전달만 하는 계층이나 이름만 다른 래퍼는 만들지 않는다.

### 불필요한 코드 금지

- 현재 인수 조건에 없는 기능을 미리 구현하지 않는다.
- 언젠가 필요할 수 있다는 이유로 인터페이스, 팩토리, 설정 옵션과 확장 지점을 만들지 않는다.
- 이미 표준 라이브러리, 플랫폼 또는 프로젝트 안에 있는 기능을 다시 만들지 않는다.
- 호출되지 않는 코드, 사용하지 않는 매개변수, 죽은 분기와 임시 호환 계층을 남기지 않는다.
- 새 의존성은 직접 구현보다 명확한 이점이 있고 유지관리 비용을 감당할 수 있을 때만 추가한다.

### 코드 스타일 일관성

- 저장소 루트의 `.editorconfig`는 에디터와 AI 도구가 공유하는 최소 표현 규칙이다. 최소한 UTF-8,
  LF, 파일 끝 개행, trailing whitespace, indent style·size를 선언한다.
- monorepo나 언어별 규칙이 다르면 파일 glob section으로 명시하며 개인 전역 EditorConfig에
  의존하지 않는다.
- 같은 언어와 계층에서 이름, import 순서, 들여쓰기, 따옴표, 파일·모듈 구성과 오류 처리 방식을
  일관되게 유지한다.
- formatter와 linter 설정을 스타일의 실행 가능한 단일 기준으로 사용하며 AI 도구별 기본 포맷을
  별도로 적용하지 않는다.
- 자동화되지 않은 선택은 승인된 style guide와 인접한 코드의 반복되는 관례를 따른다. 단 하나의
  파일이나 예외적인 오래된 코드를 전체 프로젝트 관례로 추정하지 않는다.
- 같은 문제를 해결하는 기존 패턴이 적절하면 재사용하고, 이유 없이 새로운 패턴·라이브러리·
  디렉터리 구조를 병렬로 만들지 않는다.
- 생성 코드, 외부 vendor 코드와 migration 중인 영역은 적용 범위와 예외 이유를 명시한다.
- 기존 관례가 보안, 정확성, 가독성, 접근성 또는 현재 표준을 위반하면 그대로 복제하지 않는다.
  영향 범위, 자동 migration, 호환성과 rollback을 검토해 관련 코드를 한 번에 통일한다.
- style 변경과 기능 변경은 가능한 경우 분리해 diff와 회귀 원인을 검토하기 쉽게 한다.

일관성은 잘못된 패턴을 계속 복사한다는 뜻이 아니다. 충돌 시 정확성·안전성·접근성이 기존
관례보다 우선하며, 새 기준을 선택했다면 formatter·linter·문서와 관련 코드를 함께 갱신한다.

## 함수와 API 설계

### 매개변수 기준

함수·메서드·생성자의 위치 인자와 매개변수는 기본적으로 **최대 3개**다. 4개 이상은 오류로
단정하기보다 다음 설계 문제가 있는지 확인하는 강한 리뷰 신호로 사용한다.

- 함수가 둘 이상의 책임을 가지는가
- 여러 값이 하나의 도메인 개념을 이루는가
- 호출자가 순서나 boolean 의미를 기억해야 하는가
- 일부 매개변수 조합만 유효한가
- 함수가 잘못된 계층에 있는가

권장 해결 순서:

1. 필요하지 않은 매개변수를 제거한다.
2. 함수를 책임별로 나눈다.
3. 이미 존재하는 도메인 객체에 동작을 옮긴다.
4. 함께 움직이는 값에 의미 있는 값 객체를 만든다.
5. 선택 항목이 많다면 이름 있는 옵션 객체를 사용한다.

다음과 같은 기계적 우회는 금지한다.

```text
나쁨: doWork({ a, b, c, d })
이유: 서로 무관한 네 값을 이름 없는 options 객체로 감춰 숫자만 줄임

좋음: calculateShipping(order, destination, policy)
이유: 각 인자가 명확한 도메인 역할을 가짐
```

프레임워크 콜백, 표준 인터페이스 구현, 직렬화 경계처럼 시그니처를 통제할 수 없는 경우는
예외다. 예외를 감추기 위한 래퍼를 추가하지 않고 경계 안쪽의 자체 API를 단순하게 유지한다.

### Boolean과 선택 인자

- 의미를 알 수 없는 위치 boolean 인자를 피한다.
- `render(data, true, false)` 대신 이름 있는 모드나 별도 함수를 사용한다.
- 선택 인자가 호출마다 다른 실행 경로를 만든다면 함수를 나눌지 검토한다.
- `null`이나 빈 문자열이 특별한 명령으로 동작하지 않게 한다.

### 반환값과 오류

- 반환 형식은 호출자가 분기하기 쉽고 잘못 사용할 가능성이 낮아야 한다.
- 성공과 실패를 같은 모호한 값으로 표현하지 않는다.
- 예외를 삼키거나 로그만 남기고 성공처럼 계속 진행하지 않는다.
- 외부 경계의 오류를 도메인에 맞는 오류로 변환하되 원인 추적 정보를 보존한다.

## JavaScript와 TypeScript

### 구조 분해 할당

한 번의 구조 분해에서 **4개 이상의 이름을 꺼내는 표현**은 기본적으로 피한다.

```js
// 피한다: 필요한 문맥과 값의 출처가 한꺼번에 사라진다.
const { id, name, email, role, status } = user;

// 선호: 가까운 곳에서 객체 문맥을 유지한다.
const displayName = user.name;
const canManage = user.role === "admin" && user.status === "active";
```

세 값 이하라도 별칭, 기본값과 중첩 구조 분해가 겹쳐 읽기 어렵다면 풀어서 쓴다. 반대로
React hook이나 표준 tuple처럼 구조가 짧고 관례가 분명하면 구조 분해를 사용할 수 있다.

### Spread와 rest

한 표현식, 함수 호출 또는 객체 생성에서 spread/rest 연산이 **4개 이상** 등장하면 데이터
모델이나 상태 소유권을 재검토한다.

```js
// 피한다: 우선순위와 최종 값의 출처를 추적하기 어렵다.
const config = { ...base, ...environment, ...team, ...user, ...override };

// 선호: 허용된 병합 규칙을 이름 있는 함수에서 명시한다.
const config = buildRuntimeConfig({ base, environment, override });
```

`buildRuntimeConfig`가 단순히 같은 spread를 숨기는 래퍼여서는 안 된다. 허용 필드, 우선순위,
검증과 반환 타입을 명시해야 한다.

다음 상황은 예외 후보지만 리뷰와 테스트가 필요하다.

- API 응답이나 스키마의 기계적 매핑
- immutable update가 요구되는 프레임워크 경계
- 라이브러리의 표준 rest 전달 패턴
- 생성 코드

### 객체와 상태

- 거대한 범용 `options`, `data`, `props` 객체로 의존성을 숨기지 않는다.
- 컴포넌트 props가 계속 늘면 컴포넌트 책임과 합성 구조를 점검한다.
- 여러 spread로 상태를 덧대기보다 상태 모델과 전이를 명시한다.
- 타입을 맞추기 위한 무분별한 assertion과 `any`를 사용하지 않는다.

## 시멘틱 HTML과 웹 접근성

### 의미가 먼저다

HTML 요소는 기본 스타일이 아니라 콘텐츠 의미와 사용자 동작으로 선택한다.

| 의도 | 사용할 요소 | 피할 패턴 |
|---|---|---|
| 다른 URL·문서·화면으로 이동 | `<a href="…">` | 이동을 수행하는 `<button>` 또는 클릭 `<div>` |
| 현재 화면에서 동작 실행 | `<button type="button">` | `href="#"`, 클릭 `<span>` 또는 `<div>` |
| 문단 | `<p>` | 문단처럼 보이는 여러 `<div>` |
| 강한 중요성·긴급성 | `<strong>` | 굵게 보이기 위한 `<span>`만 사용 |
| 문맥상 강조 | `<em>` | 의미 없이 스타일 클래스만 적용 |
| 제목 | `<h1>`부터 `<h6>` | 큰 글자를 만든 `<div>` 또는 단계 건너뛰기 |
| 항목 목록 | `<ul>`, `<ol>`, `<li>` | 반복되는 `<div>` 묶음 |
| 주요 탐색 | `<nav>` | 이름 없는 일반 컨테이너 |
| 독립 콘텐츠 | `<article>` | 모든 영역을 `<section>`이나 `<div>`로 처리 |
| 데이터 표 | `<table>`, `<th>`, `<caption>` | CSS grid로 표 의미 재현 |
| 입력 이름 | `<label>` | placeholder만 제공 |

`<span>`은 문장 안의 의미 없는 인라인 스타일 훅에 사용할 수 있지만, 실제 중요성이나 강조가
있으면 `<strong>` 또는 `<em>`을 선택한다. `<div>`는 의미가 없는 레이아웃 그룹에만 사용한다.

### 링크와 버튼

- 링크는 유효한 `href`를 가진다.
- 새 창을 열면 사용자에게 알리고 필요한 보안 속성을 적용한다.
- 버튼은 폼 제출 여부를 명시하기 위해 적절한 `type`을 사용한다.
- 링크와 버튼의 접근 가능한 이름만으로 목적을 이해할 수 있게 한다.
- 아이콘만 있는 버튼은 시각적으로 숨긴 텍스트나 적절한 이름을 제공한다.
- 비활성 동작을 CSS로만 막지 않고 의미와 포커스 동작을 함께 처리한다.

### 네이티브 요소 우선

- 네이티브 버튼, 링크, 입력, dialog와 details로 해결할 수 있으면 커스텀 role을 만들지 않는다.
- ARIA는 부족한 의미를 보완하며 잘못 선택한 HTML을 정당화하는 용도로 사용하지 않는다.
- 커스텀 위젯이 필요하면 해당 패턴의 키보드 조작, role, name, state와 focus 관리를 모두 구현한다.

### 문서 구조

- 페이지에는 목적을 설명하는 하나의 주 제목이 있다.
- `<h1>`부터 `<h6>`은 글자 크기를 고르는 도구가 아니며 제목 단계는 문서 구조를 따른다.
- 큰 글자가 필요하다는 이유로 heading을 사용하지 않고 CSS 또는 디자인 토큰으로 표현한다.
- 실제 제목이라면 시각적으로 작더라도 heading 의미를 유지하고 CSS로 크기를 조정한다.
- heading처럼 보이지만 section을 소개하지 않는 문구는 문맥에 맞는 `<p>`나 다른 요소를 사용한다.
- heading 단계는 스타일 때문에 건너뛰지 않는다.
- `header`, `nav`, `main`, `aside`, `footer` landmark를 의미에 맞게 사용한다.
- landmark와 section을 불필요하게 중첩하지 않는다.
- 이미지에는 목적에 맞는 대체 텍스트를 제공하고 장식 이미지는 접근성 트리에서 제외한다.

### 키보드와 포커스

- 모든 대화형 기능은 키보드만으로 사용할 수 있어야 한다.
- 보이는 포커스 표시를 제거하지 않는다.
- DOM 순서와 탭 순서가 읽기 순서와 일치해야 한다.
- 양수 `tabindex`로 순서를 강제하지 않는다.
- modal, menu와 동적 UI는 포커스 이동과 복귀를 명시적으로 검증한다.
- hover에서만 나타나는 정보는 focus와 touch에서도 접근 가능해야 한다.

### 폼과 오류

- 입력은 연결된 label과 필요 시 설명을 가진다.
- 필수 여부와 형식은 색상이나 placeholder에만 의존하지 않는다.
- 오류 메시지는 문제와 해결 방법을 설명하고 해당 입력과 연결한다.
- 제출 실패 시 오류 요약 또는 첫 오류로 사용자를 안내한다.
- 자동 완성과 입력 목적을 브라우저가 이해할 수 있게 적절한 속성을 사용한다.

## 리뷰 체크리스트

### 모든 코드

- 이름과 구조만으로 주요 의도가 읽히는가
- 한 함수나 모듈에 서로 다른 변경 이유가 섞이지 않았는가
- 현재 요구사항에 없는 코드나 추상화가 추가되지 않았는가
- 매개변수가 4개 이상인 API가 있으며 더 나은 도메인 모델이 가능한가
- 새 계층과 래퍼가 실제 책임을 갖는가
- 실패와 경계 조건을 명시적으로 처리하고 테스트했는가
- 같은 계층의 기존 코드와 naming·구성·오류 처리 스타일이 일관적인가
- 새 스타일을 도입했다면 기존 기준과의 차이, 적용 범위와 migration이 명확한가

### JavaScript와 TypeScript

- 긴 구조 분해로 객체 문맥을 잃지 않았는가
- 여러 spread의 병합 우선순위를 사람이 쉽게 추적할 수 있는가
- props와 options 객체가 책임 과다를 숨기지 않는가
- 타입 assertion, `any`, optional chaining이 설계 결함을 가리지 않는가

### 웹 UI

- 링크와 버튼이 이동과 동작의 의미에 맞는가
- 클릭 가능한 `<div>`나 `<span>`이 없는가
- 문단, 제목, 강조, 목록과 표가 시멘틱 요소로 구성됐는가
- heading이 글자 크기를 위해 선택되지 않았고 문서 개요가 논리적인가
- 폼 label, 오류, 설명과 자동 완성이 연결됐는가
- 키보드만으로 모든 흐름을 완료할 수 있는가
- 포커스 표시와 순서가 논리적인가
- 자동 접근성 검사와 실제 브라우저 검증을 수행했는가

## 자동화 전략

숫자 규칙을 모든 언어에 동일한 정규식으로 강제하지 않는다. 언어별 AST 기반 린터를
사용하고, 기계적으로 판단하기 어려운 관심사 분리와 추상화 품질은 리뷰에서 검증한다.

| 기준 | 권장 검증 |
|---|---|
| 인자·매개변수 최대 3개 | 언어별 복잡도 린터 + 예외 allowlist |
| 사용하지 않는 코드와 인자 | 컴파일러·린터 |
| 문자셋·줄바꿈·파일 끝 개행·기본 들여쓰기 | `.editorconfig` + 저장소 설정 검사 |
| 들여쓰기·따옴표·import 순서 등 스타일 | EditorConfig와 충돌 없는 formatter·linter + CI |
| 구조 분해·spread 복잡도 | ESLint 커스텀 규칙 또는 AST 검사 |
| 클릭 가능한 비시멘틱 요소 | JSX/템플릿 접근성 린터 |
| label, ARIA, heading 기본 오류 | 접근성 린터와 자동 브라우저 검사 |
| 키보드·포커스·스크린 리더 흐름 | E2E + 수동 검증 |
| 관심사 분리와 불필요한 추상화 | 코드 리뷰 + Ponytail 계열 감사 |

프론트엔드의 구체적인 실행 시점과 훅 구성은
[프론트엔드 도구와 훅](frontend-tooling-and-hooks.md)을 따른다.

자동 규칙의 예외에는 이유, 적용 범위와 제거 조건을 가까운 설정 또는 ADR에 기록한다.
