# AI 협업 SDLC

상태: 제안

## 목표

AI가 코드를 빠르게 생성하는 것보다, 요구사항에서 운영까지 추적 가능하고 재현 가능한
흐름을 만드는 것이 목적이다. 모든 단계에는 입력, 산출물, 품질 게이트와 책임자가 있다.

각 단계의 탐색·설명 깊이는 [토큰 예산별 실행 프로파일](token-budget-profiles.md)에 따라
조정할 수 있다. 단, 프로파일과 관계없이 요구사항 충족, 보안 승인과 변경 관련 필수 검증은
동일하다.

```text
clone·onboarding → 요청 → 발견 → 요구사항·Eval → 설계 → 계획 → 구현 → 검증 → 리뷰 → 릴리스
                         ▲                                                   │
                         └──────── 운영·학습과 regression 환류 ──────────────┘
```

## 0. Clone과 개발 준비

사용자별 AI 도구와 전역 환경이 다르다는 전제에서 읽기 전용 진단, 승인된 프로젝트 로컬 설치와
`validate`를 실행한다. 공통 정책, hook, 보안 검사와 필수 Eval grader가 연결되지 않았다면
코드 생성을 시작하지 않고 미준비 항목을 보고한다.

작업 대상이 공통 환경 자체라면 upstream maintenance, 회사·프로젝트 적용이라면 downstream
adoption으로 분류한다. 역할별 변경 경계와 완료 조건은
[유지보수와 도입 모델](adoption-and-maintenance-model.md)을 따른다.

## 1. 발견과 요구사항

입력은 사용자의 문제와 제약이다. AI는 모호한 요청을 그대로 구현하지 않고 사용자 흐름,
범위, 비기능 요구사항과 인수 조건으로 구체화한다.

산출물:

- 고유 ID를 가진 요구사항
- 포함 및 제외 범위
- 검증 가능한 인수 조건
- 미해결 질문과 가정
- 요구사항에서 파생한 대표 Eval task와 성공 outcome

게이트: 구현 결과가 맞는지 판단할 수 없는 요구사항은 개발 준비 상태가 아니다.

새 프로젝트에서는 이 단계 전에 하네스 구성 인터뷰를 실행할 수 있다. 인터뷰는 기술 스택,
위험도, 배포 대상과 필요한 품질 게이트를 질문하지만, 답변은 반드시 버전 관리되는 설정과
요구사항으로 변환한다. 채팅 기록 자체는 구성의 단일 진실 원천이 아니다.

사람의 의도와 성공 기준을 구조화해야 하면 [프롬프트 템플릿](prompt-templates.md)을 사용한다.
확정된 내용은 요구사항과 ADR로 옮기고 반복 실행·검증은 하네스가 담당한다.

public 웹·앱이면 [웹서비스 Production 준비 가이드](web-service-production-readiness.md)의 개인정보,
아동, 국외이전·위탁, 위치, 광고, 전자상거래, subscription, 사용자 제공 AI, paid API와 database
적용 여부를 `yes | no | TBD`로 분류한다.

## 2. 설계

기존 시스템과 제약을 조사하고 최소 두 가지 대안이 의미가 있을 때 비교한다. 중요한 결정은
ADR로 남기고, 작은 로컬 변경에는 불필요한 설계 문서를 만들지 않는다.

산출물:

- 변경되는 컴포넌트와 인터페이스
- 데이터 및 보안 영향
- 테스트와 롤백 전략
- 필요한 ADR

게이트: 고위험 변경은 구현 전에 사람의 설계 승인을 받는다.

## 3. 계획

작업을 독립적으로 검증 가능한 작은 단위로 나눈다. 각 단위는 관련 요구사항, 변경 파일,
테스트 방법과 완료 조건을 가진다.

게이트: 계획에 검증과 문서 갱신이 포함되어야 한다.

## 4. 구현

- 작은 diff와 짧은 피드백 주기를 유지한다.
- 기존 코드와 사용자 변경을 보존한다.
- 코드 변경과 관련 테스트를 함께 작성한다.
- 의존성을 추가할 때 필요성, 라이선스, 유지관리 상태와 보안 영향을 검토한다.
- AI·generator가 만든 code도 출처·license를 확인하며 unresolved match를 재작성만으로 우회하지 않는다.
- API 변경은 승인된 contract와 동기화하고 breaking change·문서 공개 범위를 검증한다.
- 생성물에 비밀정보, 개인 데이터, 임시 우회 코드를 포함하지 않는다.
- 언어와 프레임워크에 관계없이 [코드 품질 표준](code-quality-standards.md)을 적용한다.
- 웹 UI는 동작 전에 시멘틱 HTML 구조와 키보드 사용성을 먼저 확인한다.

게이트: 포맷, 정적 분석, 단위 테스트가 로컬에서 통과해야 한다.

REST·GraphQL·gRPC·event API는 선택한 contract의 syntax·implementation drift와 backward compatibility를
CI에서 검사한다. Swagger UI 같은 문서 화면 생성은 authorization·contract test를 대체하지 않는다.

승인된 JavaScript·TypeScript 프론트엔드 프로파일에서는 AI의 변경 묶음이 끝날 때 변경 파일에
Prettier와 ESLint 자동 수정을 적용하고, 커밋 시 staged 파일을 Husky 훅으로 다시 검사한다.
개발환경이 확정되기 전에는 Husky를 설치하지 않는다. 상세 정책은
[프론트엔드 도구와 훅](frontend-tooling-and-hooks.md)을 따른다.

## 5. 검증

변경 위험에 비례해 다음 계층을 적용한다.

1. 포맷과 린트
2. 타입 및 정적 분석
3. 단위 테스트
4. 통합 테스트
5. 핵심 사용자 흐름 E2E
6. 접근성, 성능, 보안 검사
7. 필요 시 실제 UI의 시각 검증

테스트 실패를 단순 재실행으로 숨기지 않고 제품 결함, 테스트 결함, 환경 문제로 분류한다.

하네스, prompt, skill, model과 반복 로직을 변경했다면 [Eval 전략](evaluation-strategy.md)의
고정 suite에서 이전 baseline과 비교한다.

웹 UI 검증에는 최소한 시멘틱 요소, 접근 가능한 이름, 키보드 조작, 포커스 표시와 제목
계층 검사를 포함한다. 자동 접근성 검사는 수동 키보드·스크린 리더 검토를 완전히 대체하지 않는다.

게이트: 관련 검증이 통과하고, 실행하지 못한 검사는 이유와 위험을 기록해야 한다.

## 6. 리뷰

리뷰 순서는 요구사항 충족, 정확성, 보안과 데이터 안전, 회귀 위험, 운영 가능성,
유지보수성 순이다. 스타일 선호는 자동화된 규칙에 맡긴다.

게이트:

- 필수 CI 통과
- 필요한 승인 획득
- 미해결 고위험 리뷰 의견 없음
- 문서와 변경 기록 동기화

AI가 작성한 변경도 사람이 작성한 변경과 동일한 기준을 적용한다.

## 7. 릴리스

- 버전과 변경 기록을 작성한다.
- 환경 설정과 마이그레이션 순서를 확인한다.
- 배포 전 백업과 롤백 가능성을 확인한다.
- 점진 배포가 가능하면 작은 범위부터 노출한다.
- 운영 배포는 사람의 명시적 승인을 요구한다.
- 적용되는 개인정보·소비자·위치·광고·AI 법률과 처리방침·동의·보존·파기 evidence를 확인한다.
- admin·object authorization, rate limit·비용, secret, DB exposure와 backup restore 결과를 확인한다.

게이트: 배포 후 확인할 지표와 롤백 조건이 정의되어야 하며 Production readiness 필수 항목에 `TBD`나
미검증 고위험 통제가 없어야 한다.

## 8. 운영과 학습

로그, 지표, 오류, 사용자 피드백을 관찰하고 예상 결과와 비교한다. 사고와 반복되는 리뷰
지적은 개인의 기억에 맡기지 않고 테스트, 표준 또는 스킬 개선으로 환류한다.

산출물:

- 배포 후 검증 결과
- 장애 회고 및 후속 작업
- 신규 또는 변경 요구사항
- 하네스와 스킬 개선 제안

## 변경 유형별 최소 경로

| 변경 유형 | 최소 필수 단계 |
|---|---|
| 문서 오탈자 | 구현 → 링크·형식 검증 → 리뷰 |
| 작은 버그 수정 | 요구사항 → 구현 → 회귀 테스트 → 리뷰 |
| 새 기능 | 요구사항 → 설계 → 계획 → 구현 → 전체 검증 → 리뷰 → 릴리스 |
| 인증·결제·개인정보 | 새 기능 경로 + 보안 리뷰 + 명시적 승인 |
| 데이터 마이그레이션 | 설계 + 백업·롤백 검증 + 단계적 릴리스 |
| 긴급 수정 | 축소된 사전 게이트 + 승인 + 배포 직후 전체 검증과 회고 |

## 추적성

권장 연결 구조는 다음과 같다.

```text
요구사항 ID → 이슈/작업 → 커밋/PR → 테스트 → 릴리스 → 운영 지표
```

PR 또는 변경 설명에는 관련 요구사항 ID, 주요 결정, 검증 결과와 남은 위험을 기록한다.
