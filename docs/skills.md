# 스킬 체계

상태: 제안

## 정의

스킬은 AI에게 특정 역할을 부여하는 이름이 아니라, 반복 가능한 작업을 수행하는
**버전 관리된 실행 계약**이다. 한 스킬은 입력, 절차, 산출물, 검증 방법과 안전 경계를 가진다.

## 스킬 계약

각 `.ai/skills/<name>/SKILL.md`는 최소한 다음을 포함한다.

- 목적과 사용해야 하는 상황
- 사용하지 말아야 하는 상황
- 필요한 입력과 사전 조건
- 순서가 명확한 작업 절차
- 생성하거나 변경하는 산출물
- 완료 조건과 검증 명령
- 네트워크, 비밀정보, 배포 등 권한 경계
- 실패 시 중단 및 사용자 확인 조건

## 권장 분류

### 필수 기반 스킬

모든 프로젝트에 제공한다.

| 스킬 | 책임 |
|---|---|
| `requirements` | 요청을 검증 가능한 요구사항과 인수 조건으로 변환 |
| `architecture` | 제약, 대안, 결정과 영향 범위를 기록 |
| `implementation` | 작은 변경 단위로 구현하고 기존 규칙 준수 |
| `testing` | 테스트 전략 수립, 실행, 실패 원인 분류 |
| `code-review` | 정확성, 보안, 회귀 위험 중심의 리뷰 수행 |
| `documentation` | 코드와 의사결정에 맞게 문서 동기화 |
| `security-check` | 비밀 노출, 의존성, 입력 검증과 권한 점검 |
| `release` | 변경 기록, 버전, 배포 전후 검증 수행 |
| `code-quality` | 가독성, 관심사 분리, 불필요한 코드와 API 복잡도 점검 |

### 웹 애플리케이션 권장 스킬

| 스킬 | 책임 |
|---|---|
| `frontend` | 접근성, 반응형 UI, 디자인 시스템 준수 |
| `backend` | API 계약, 오류 처리, 관측 가능성 준수 |
| `database-change` | 마이그레이션, 롤백, 데이터 호환성 검토 |
| `e2e-testing` | 핵심 사용자 흐름을 실제 실행 환경에서 검증 |
| `performance` | 측정 기준에 따른 성능 예산 점검 |
| `deployment` | 환경별 배포와 롤백 절차 수행 |

`code-quality`와 `frontend` 스킬은 [코드 품질 표준](code-quality-standards.md)을 공통 계약으로
참조한다. 도구별 디자인 스킬이 이 기준을 약화할 수 없다.

### 선택 스킬

프로젝트 특성에 따라 활성화한다. 예: 모바일, AI 기능, 데이터 파이프라인, 인프라,
문서 발행, 디자인 생성. 선택 스킬은 기반 스킬의 안전 및 검증 규칙을 약화할 수 없다.

## 설치와 버전 관리

- 저장소 안의 프로젝트 스킬을 전역 설치 스킬보다 우선한다.
- 외부 스킬은 출처와 정확한 버전 또는 Git 커밋을 잠금 파일에 기록한다.
- 외부 스킬을 업데이트할 때 변경 내용을 리뷰하고 검증 프로젝트에서 먼저 실행한다.
- 같은 이름의 도구별 스킬은 공통 계약에 동일한 산출물을 만들어야 한다.
- 전역 개인 스킬은 생산성 향상에 사용할 수 있지만 필수 워크플로의 전제 조건이 될 수 없다.
- 설치 후보는 [공급망 보안 게이트](supply-chain-security.md)를 통과해야 한다.
- 선호 도구의 AI 환경별 적용 방식은 [도구 호환성 표](tool-compatibility.md)를 따른다.

## 플러그인 정책

플러그인은 스킬과 분리해 관리한다. 스킬은 작업 방법이고 플러그인은 외부 기능 제공자다.

- `필수`: 없으면 프로젝트 핵심 작업을 수행할 수 없음
- `권장`: 자동화 수준을 높이지만 수동 대체 절차가 있음
- `개인`: 개인 취향이며 팀 결과에 영향을 주지 않음

필수 및 권장 플러그인은 지원 도구별 이름, 버전, 권한, 대체 절차를 `tools.yaml`에 기록한다.
플러그인에서 얻은 결과도 테스트나 리뷰 같은 저장소의 품질 게이트를 통과해야 한다.

공통 배포의 우선순위는 `감사된 Markdown 스킬 → 감사된 로컬 스크립트 → 훅 또는 플러그인
런타임 → 외부 MCP` 순이다. 뒤로 갈수록 권한과 공격 표면이 커지므로 더 강한 검토가 필요하다.

## 품질 기준

- 스킬 문서는 하나의 주요 책임을 가진다.
- 도구 이름보다 결과와 검증 조건을 중심으로 작성한다.
- 모호한 표현 대신 실행 가능한 명령이나 체크리스트를 제공한다.
- 예시 프로젝트에서 성공 경로와 실패 경로를 모두 시험한다.
- 중복 스킬은 통합하고, 특화 스킬은 기반 스킬을 참조한다.
