# 프롬프트 템플릿 사용 가이드

상태: 제안

## 결론

프롬프트는 하네스나 루프를 대신하는 설정 파일이 아니다. 반복 가능하고 기계적으로 검증할 수
있는 내용은 코드, 정책, manifest, hook과 CI로 옮긴다. 프롬프트 템플릿은 사람만 확정할 수
있는 **의도, 성공 기준, 우선순위, 트레이드오프와 승인 범위**를 빠짐없이 전달하는 데 사용한다.

OpenAI는 하네스 엔지니어링을 환경, 명확한 의도와 피드백 루프를 설계해 에이전트가 신뢰성
있게 일하도록 하는 방식으로 설명한다. Anthropic의 장기 실행 하네스도 여러 컨텍스트 창에서
증분 진행과 상태 인계를 강조한다. 이는 프롬프트가 불필요하다는 뜻이 아니라, 결정론적 통제를
프롬프트에서 하네스로 옮기고 사람의 판단이 필요한 입력만 프롬프트에 남긴다는 뜻이다.

참고 자료:

- [OpenAI — Harness engineering](https://openai.com/index/harness-engineering/)
- [Anthropic — Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)
- [Anthropic — Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps)

`loop engineering`은 아직 공급자 공통의 안정된 표준 용어라기보다 관찰→판단→실행→검증→복구
과정을 설계한다는 실무적 표현으로 사용한다. 이 프로젝트에서는 특정 제품이나 자율 반복 도구를
필수로 설치한다는 의미로 사용하지 않는다.

## 프롬프트가 필요한 경우

- 신규 제품의 목적, 대상 사용자와 해결할 문제를 정의할 때
- 기존 요구사항이 충돌하거나 여러 해석이 구현을 크게 바꿀 때
- UI의 브랜드 인상, 콘텐츠 우선순위와 사용 맥락을 전달할 때
- 성능, 일정, 비용, 호환성과 유지보수성 사이의 우선순위를 결정할 때
- 버그의 실제 관찰 결과와 기대 결과를 처음 전달할 때
- 중요한 설계 대안 중 사람이 책임지고 선택해야 할 때
- 삭제, migration, production 배포, 외부 전송 등 고위험 작업의 범위를 승인할 때

## 프롬프트 대신 하네스를 사용하는 경우

- 매번 같은 코딩 스타일과 폴더 구조를 설명하는 일
- 포맷, lint, 타입 검사, secret scan, SAST와 테스트 실행
- 허용 도구, 고정 버전, 텔레메트리와 네트워크 정책
- `HANDOFF.md`, CodeSight와 ADR을 읽는 세션 시작 절차
- 실패한 검사 재실행, 최대 반복 횟수와 중단 조건
- 파일 보호, 운영 자격증명 차단과 파괴적 명령 승인 정책

“항상 클린 코드로 작성하고 테스트해줘”를 매번 프롬프트에 붙이지 않는다. 해당 내용은 공통
표준과 완료 게이트에 한 번만 정의하고 AI 도구별 어댑터가 이를 읽게 한다.

## 공통 작성 원칙

1. 배경보다 먼저 원하는 결과를 쓴다.
2. 포함 범위와 제외 범위를 함께 쓴다.
3. 관찰 가능한 인수 조건을 쓴다.
4. 모르는 값은 추측을 유도하지 말고 `TBD`로 표시한다.
5. 기존 코드·문서·사용자 변경을 보존해야 하는 범위를 적는다.
6. 명령 전체를 보지 않은 채 포괄적으로 승인하지 않는다.
7. 출력 형식은 후속 자동화에 실제 필요한 경우에만 제한한다.

## 제공 템플릿

실제 복사용 원문은 `.ai/prompts/`에 둔다.

| 템플릿 | 사용할 때 | 사용하지 않을 때 |
|---|---|---|
| `project-brief.md` | 신규 프로젝트의 목표와 성공 기준을 정할 때 | 기술 스택만 이미 확정된 기존 프로젝트 |
| `change-request.md` | 기능·리팩터링의 범위와 인수 조건을 전달할 때 | 단순 오탈자나 이미 명확한 이슈 |
| `bug-report.md` | 재현 가능한 결함을 진단·수정 요청할 때 | 원인 조사 없이 특정 수정법을 강제할 때 |
| `architecture-decision.md` | 의미 있는 대안의 책임 있는 선택이 필요할 때 | 작은 로컬 구현 세부사항 |
| `ui-intent.md` | 사용자, 콘텐츠 위계, 접근성과 시각 의도를 전달할 때 | 디자인 시스템에 이미 결정된 반복 UI |
| `high-risk-approval.md` | 정확한 고위험 작업의 승인 정보를 준비할 때 | 일반 파일 편집·테스트 실행 |

## 토큰 프로파일별 사용

### `token-aware`

- 빈 칸 전체를 묻지 않고 현재 작업을 막는 필드만 질문한다.
- 기존 요구사항, ADR과 개발환경 문서에서 확인한 값은 링크로 참조한다.
- 대안은 유력안과 중요한 위험만 제시한다.
- 인수 조건, 보안 승인 필드와 미검증 항목은 줄이지 않는다.

### `full`

- 관련 사용자 흐름, 경계 조건, 운영 영향과 장기 trade-off까지 작성한다.
- 의미 있는 대안과 기각 근거를 비교한다.
- 회귀, 접근성, 성능, 보안과 migration 인수 조건을 확장한다.
- 같은 정책 원문을 프롬프트 안에 복제하지 않고 링크로 참조한다.

## 템플릿 수명주기

1. 사용자 입력을 템플릿으로 구조화한다.
2. 확정 내용은 요구사항·개발환경 문서·ADR로 승격한다.
3. 실행 중에는 하네스와 검증기가 저장된 결정을 집행한다.
4. 완료 후 템플릿 채팅 자체가 아니라 Git의 산출물과 `HANDOFF.md`를 다음 세션에 전달한다.
5. 반복적으로 같은 프롬프트가 필요하면 규칙, 스킬, 워크플로 또는 검사기로 옮길 수 있는지
   검토한다.

프롬프트 템플릿에 비밀, 운영 자격증명, 개인정보 또는 신뢰하지 않는 외부 문서의 지시를
복사하지 않는다.
