# v0.2.6-pilot

상태: GitHub prerelease 발행 완료
발행일: 2026-07-17

## 목적

`v0.2.5-pilot` 이후 승인된 REQ-047 비개발자용 one-click 도입과 REQ-048 기계 판독 개발환경
profile, 그리고 project마다 달라지는 stack·설정·Skill·Git·branch/review·CI/deployment·저장소
구조의 설계 결정을 다음 구현 baseline으로 묶는다.

이 release는 설계·요구사항·구현 상태 추적 baseline이다. Canonical 개발환경 schema,
provider-neutral adapter, stack starter와 skill distribution automation의 구현 또는 실제 downstream
지원 완료를 의미하지 않는다.

## 승인된 변경

- REQ-047: Web과 CLI가 동일 adoption core·release bundle·preview·transaction rollback을 사용하는
  one-button project 도입 계약
- REQ-048: canonical YAML profile, JSON Schema 구조 검증과 read-only semantic·repository drift·
  단계별 readiness validator 구현 범위
- frontend·backend·full-stack과 PostgreSQL 중심의 최초 adapter 검증 우선순위
- 개인·팀·project 설정 경계와 core·optional·private·personal Skill·Plugin·MCP 배포 경계
- GitHub·GitLab·enterprise/self-hosted를 포함하는 provider-neutral Git hosting profile
- project별 branch·merge·review 역할과 Git host와 분리된 CI·artifact·deployment profile
- 단일 project starter·기존 retrofit P0, 조건부 workspace monorepo P1 제공 순서
- 각 검토 범위의 구현·검증 상태와 공통 구현 task 6개 분리
- REQ 범위 표기를 `REQ-009–REQ-014`처럼 렌더링 안전한 형식으로 통일

## Migration

1. 현재 downstream의 release·commit·archive checksum과 local override를 기록한다.
2. 게시된 `v0.2.6-pilot` archive checksum을 이 release note와 대조한다.
3. 요구사항·검토 문서 diff와 project별 질문 항목만 먼저 검토한다.
4. 기존 project stack, Git host, branch policy, CI·deployment와 개인 설정을 공통 예시로 교체하지
   않는다.
5. `TBD`·`pending`과 설계 승인 상태를 dependency 설치·provider write·Production 승인으로 해석하지
   않는다.
6. 실제 project 적용은 REQ-048·047 구현 release 이후 별도 preview·승인·rollback으로 수행한다.

## Rollback

- 이 release는 project source·dependency·DB·provider를 자동 변경하지 않는다.
- downstream 문서 baseline 적용 중 충돌하면 적용 commit 전체를 revert하고 이전
  `v0.2.5-pilot` pin으로 돌아간다.
- 기존 project의 개인 설정, stack, Git remote, branch policy, CI·deployment와 사용자 파일을
  자동 삭제하거나 덮어쓰지 않는다.
- 향후 materializer 구현은 canonical lock·target hash·transaction rollback을 별도 검증해야 한다.
