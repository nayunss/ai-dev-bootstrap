# AI 개발 하네스 구성

상태: 공통 하네스 reference 구현
갱신일: 2026-07-19

## 결론

하네스 구성에 특정 플러그인은 필요하지 않다. 하네스는 플러그인이 아니라 다음 요소가 함께
이루는 **저장소 수준의 실행 계약**이다.

- 프로젝트와 AI가 따라야 할 규칙
- 도구가 규칙을 발견하는 진입점
- 재사용 가능한 스킬과 워크플로
- 허용된 도구, 버전과 권한
- 로컬 및 CI 품질 게이트
- 요구사항, 결정과 변경 이력

플러그인은 이를 생성하거나 자동으로 불러오고 실행하는 편의 계층이다. 플러그인이 없어도
사람과 AI가 문서를 읽고 같은 검증 명령을 실행할 수 있어야 한다.

## FAQ

### 특정 플러그인을 설치해야 하네스를 구성할 수 있는가

아니다. Markdown 문서, 설정 파일, 셸 스크립트와 CI만으로도 하네스를 구성할 수 있다.
하네스 플러그인은 프로젝트 분석, 파일 생성과 감사 작업을 자동화하는 선택 사항이다.

전용 플러그인이 생성한 결과도 저장소에 일반 파일로 남아야 하며, 플러그인을 제거한 뒤에도
다른 AI 도구와 개발자가 내용을 이해하고 실행할 수 있어야 한다.

### 공통 AI 개발환경에 하네스가 필수인가

최소 하네스는 필수다. 여기서 필수라는 것은 특정 제품이나 플러그인을 반드시 설치한다는
뜻이 아니다. 공통 규칙, 도구별 진입점, 승인된 도구 목록과 검증 명령이 저장소에 존재해야
한다는 뜻이다.

전문 에이전트, MCP, 자동 훅, 복잡한 오케스트레이션은 선택 사항이며 실제 필요가 검증된
뒤에 추가한다.

### AI에게 프롬프트로 질문해 하네스를 적용할 것인가

초기 구성 인터뷰에는 AI 프롬프트를 사용한다. AI는 기술 스택, 애플리케이션 유형, 배포 환경,
데이터 민감도와 품질 수준을 질문하고 적합한 프로필을 추천할 수 있다.

하지만 최종 적용은 자유형 응답으로 끝내지 않는다. 답변을 구조화된 설정 파일로 변환하고,
생성될 파일과 설치 항목을 사용자에게 보여준 뒤 승인된 변경만 적용한다. 적용 후에는 로컬
검증과 CI가 같은 결과를 내야 한다.

### 권장 기본 정책은 무엇인가

- 프롬프트: 선택지 설명과 설정 초안 작성에 사용
- 구성 파일: 최종 선택과 현재 상태의 단일 진실 원천
- Git diff: 사람의 검토와 승인 수단
- `bootstrap`: 승인된 설정을 결정론적으로 적용
- `validate`와 CI: 개발자와 AI 도구에 관계없이 동일한 상태를 검증
- 플러그인: 반복 작업을 줄이는 선택적 어댑터

## 공통 환경에서 하네스가 필요한가

**최소 하네스는 필요하다.** 하네스가 전혀 없으면 공통화하려는 규칙, 스킬, 폴더 구조와
검증 절차가 다시 각 개발자의 전역 설정과 채팅 습관에 흩어진다.

하지만 처음부터 복잡한 멀티에이전트 플랫폼이나 대규모 플러그인 묶음은 필요하지 않다.
초기 필수 범위는 다음이면 충분하다.

```text
최소 공통 하네스
├── AGENTS.md / CLAUDE.md     도구별 발견 어댑터
├── .ai/standards/            공통 개발 규칙
├── .ai/manifests/            지원 도구와 승인 버전
├── docs/requirements.md      요구사항의 단일 기록
├── HANDOFF.md                최근 세션의 재개 지점
├── scripts/bootstrap         환경 구성
└── scripts/validate          동일 상태와 품질 검증
```

스킬, 전문 에이전트, MCP와 자동 훅은 실제 반복 작업이나 위험이 확인될 때 점진적으로 추가한다.

## 플러그인이 하는 일과 하지 않아야 하는 일

하네스 생성·관리 플러그인은 다음 작업을 편리하게 만들 수 있다.

- 프로젝트를 분석해 필요한 규칙과 스킬 후보 추천
- `AGENTS.md`, `CLAUDE.md`와 스킬 골격 생성
- 훅, 에이전트와 MCP 설정 제안
- 누락된 설정과 중복 규칙 점검

다음 책임을 플러그인에만 맡기면 안 된다.

- 팀 정책의 단일 진실 원천
- 설치된 스킬과 플러그인의 승인 목록
- 품질 게이트와 CI
- 보안 및 텔레메트리 정책
- 프로젝트 고유 요구사항과 설계 결정

플러그인을 제거해도 저장소의 하네스가 온전히 이해되고 실행되어야 한다.

## 프롬프트로 구성할 것인가

권장 방식은 **대화형 인터뷰 + 결정론적 생성 + 검증**이다.

```text
AI 인터뷰
   │  기술 스택·배포·위험도·팀 규칙 질문
   ▼
구조화된 구성 파일
   │  사람이 diff를 검토하고 승인
   ▼
하네스 파일 생성
   │
   ▼
로컬 validate + CI
```

AI에게 “이 프로젝트에 좋은 하네스를 알아서 구성해줘”라고 매번 요청하는 방식은 권장하지
않는다. 모델과 도구에 따라 결과가 달라지고, 이전 결정의 근거와 설치 버전을 재현하기 어렵다.

프롬프트는 다음 범위에만 사용한다.

- 사용자가 모르는 선택지를 설명한다.
- 프로젝트 상황에 맞는 프로필을 추천한다.
- 설정 초안과 변경 이유를 작성한다.
- 기존 하네스의 누락과 충돌을 감사한다.

제품 의도, 변경 요청, 버그 재현, 설계 선택과 고위험 승인처럼 사람의 판단이 필요한 입력은
[프롬프트 템플릿 사용 가이드](prompt-templates.md)의 최소 템플릿을 활용한다. 반복 제어,
권한과 검증은 템플릿이 아니라 하네스와 루프에 유지한다.

하네스와 반복 루프의 품질은 agent의 자기 보고가 아니라 고정된 Eval task와 outcome grader로
측정한다. 하네스·스킬·프롬프트·모델 변경 전후에는 [Eval 전략](evaluation-strategy.md)의
regression suite와 baseline을 비교한다.

최종 선택은 `docs/development-environment.profile.yaml`에 저장하고 Git diff로 검토한다. 이 profile은
이미 구현된 [Development environment profile schema](schemas/development-environment-profile.schema.json)와
semantic·repository·readiness validator를 사용한다. AI 도구 선택은 별도의 adapter preview와
`.ai/manifests/adapters.lock.json`으로 추적하므로 profile 안에 임의의 `aiTools` field를 추가하지
않는다. 실제 예시는 `evals/fixtures/development-profile/`의 네 topology fixture를 따른다.

## 권장 구성 흐름

### 저장소 clone 후 온보딩

1. 운영체제, AI 도구, 런타임과 현재 설정을 읽기 전용으로 진단한다.
2. manifest·lockfile과 비교해 필수·선택·충돌·미지원 항목을 보여준다.
3. 설치 파일, 버전, 해시, 권한, 네트워크와 전역 변경 여부를 사용자에게 제시한다.
4. 승인된 항목만 프로젝트 로컬에 idempotent하게 설치한다.
5. Codex·Claude Code·GitHub Copilot 어댑터가 동일한 공통 정책과 명령을 참조하는지 검사한다.
6. `validate`, security gate와 필수 regression Eval을 통과한 뒤 개발을 시작한다.

개인 전역 설정은 자동으로 덮어쓰지 않는다. 전역 변경이 필요한 선택 기능은 별도 승인과 제거
절차를 제공한다.

### 새 프로젝트

1. `bootstrap`을 실행한다.
2. 기본 프로필을 선택하고 필요한 최소 질문에 답한다.
3. AI는 답변을 구조화된 설정과 요구사항 초안으로 변환한다.
4. 사용자가 생성될 파일, 도구, 권한과 네트워크 접근을 검토한다.
5. 승인된 항목만 프로젝트 범위에 설치한다.
6. `validate`를 실행하고 결과를 커밋한다.

### 새 세션

1. 루트 `HANDOFF.md`를 읽는다.
2. 현재 `git status`와 handoff에 기록된 변경 파일을 대조한다.
3. 관련 요구사항, 정책과 ADR 원문을 확인한다.
4. 완료됐다고 기록된 검증 중 현재 작업에 중요한 항목을 재확인한다.
5. 다음 시작점부터 작업하고 종료 전에 handoff를 다시 갱신한다.

### 기존 프로젝트

1. 읽기 전용으로 기술 스택, 기존 AI 설정과 CI를 조사한다.
2. 현재 상태와 표준의 차이만 보고한다.
3. 기존 규칙을 덮어쓰지 않고 병합 계획을 만든다.
4. 작은 단계로 적용하고 각 단계에서 기존 테스트와 `validate`를 실행한다.

## 구성 프로필

처음에는 프로필을 적게 유지한다.

| 프로필 | 용도 | 기본 게이트 |
|---|---|---|
| `minimal` | 실험, 작은 도구, 문서 프로젝트 | 형식, 비밀 검사, 기본 테스트 |
| `web-application` | 일반적인 프론트엔드·백엔드 웹 앱 | 린트, 타입, 단위·통합·E2E, 접근성 |
| `high-risk-web` | 인증, 결제, 개인정보 또는 중요한 데이터 | 웹 앱 게이트 + 보안 리뷰, 마이그레이션·롤백 |

프로필은 기본값일 뿐이며 프로젝트 고유 설정이 명시적으로 확장한다.

이 기능·위험 프로필과 별도로 AI 실행에는 `token-aware`와 `full` 토큰 예산 프로파일을
선택할 수 있다. 토큰 프로파일은 분석과 설명의 폭만 조정하며 보안과 필수 품질 게이트를
낮추지 않는다. 상세 기준은 [토큰 예산별 실행 프로파일](token-budget-profiles.md)을 따른다.

## 도구별 적용

### Codex

- `AGENTS.md`에서 공통 코어와 현재 프로젝트 설정을 읽도록 한다.
- Codex 전용 플러그인이 없어도 핵심 하네스가 동작해야 한다.
- 공식 또는 승인된 플러그인은 스킬 발견과 자동화 향상에만 사용한다.

### Claude Code

- `CLAUDE.md`에서 같은 공통 코어와 프로젝트 설정을 읽도록 한다.
- 하네스 생성 플러그인은 초기 추천이나 감사에 선택적으로 사용할 수 있다.
- Claude 전용 훅과 서브에이전트 설정은 공통 계약의 어댑터로 취급한다.

### GitHub Copilot

- `.github/copilot-instructions.md`가 같은 공통 코어와 project profile을 참조하도록 한다.
- Copilot을 선택한 downstream에만 adapter를 materialize한다.
- Native hook parity가 없는 동작은 hosted CI와 공통 validator가 fail-closed로 검증한다.

### 다른 AI 도구

- `AGENTS.md` 또는 해당 도구의 지침 파일을 통해 같은 공통 코어를 연결한다.
- 도구에 스킬이나 에이전트 기능이 없으면 문서화된 워크플로를 순차 실행한다.
- 핵심 완료 조건은 도구 기능이 아니라 저장소의 검증 명령으로 판단한다.

## 채택 기준

하네스 기능을 필수로 승격하려면 다음을 만족해야 한다.

- 둘 이상의 실제 프로젝트에서 반복되는 필요가 확인됐다.
- Codex·Claude Code·GitHub Copilot에서 같은 핵심 결과를 낼 수 있다.
- 플러그인 없이 수행 가능한 대체 절차가 있다.
- 설치 항목이 공급망 보안 게이트를 통과했다.
- 로컬과 CI에서 상태를 자동 검증할 수 있다.

이 기준을 충족하지 못한 기능은 프로젝트 선택 옵션이나 실험 항목으로 유지한다.
