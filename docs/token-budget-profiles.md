# 토큰 예산별 실행 프로파일

상태: 제안

## 결론

프로젝트의 보안과 완료 기준은 하나로 유지하되, AI가 작업을 이해하고 설명하는 방법은 사용
가능한 토큰에 맞춰 두 가지 프로파일로 제공한다.

- `token-aware`: 제한된 토큰으로 필요한 결과와 검증에 집중한다.
- `full`: 충분한 토큰으로 더 넓은 탐색, 대안 비교와 심층 검증을 수행한다.

두 프로파일의 차이는 **분석 폭과 산출물 밀도**다. 보안 통제, 사용자 승인, 요구사항 충족과
변경에 직접 관련된 테스트의 통과 기준은 동일하다.

## 선택 방식

새 프로젝트나 새 세션에서 저장된 기본값이 없으면 다음처럼 한 번만 짧게 확인한다.

> 이번 작업은 토큰 절약형(`token-aware`)과 충분한 분석형(`full`) 중 어느 방식으로
> 진행할까요? 답이 없으면 토큰 절약형으로 시작합니다.

사용자가 정확한 토큰 수를 알 필요는 없다. “짧게”, “비용을 아껴서”, “충분히 검토”,
“꼼꼼하게” 같은 의도를 프로파일로 매핑할 수 있다. 작업 중 프로파일을 바꿀 수 있으며,
변경 전까지 완료된 검증을 불필요하게 반복하지 않는다.

프로젝트 기본값은 향후 `.ai/project.yaml`에 다음과 같이 기록할 수 있다.

```yaml
aiExecution:
  defaultProfile: token-aware
  allowSessionOverride: true
  requiredGates:
    - security
    - relevant-tests
    - handoff
```

설정 파일이 구현되기 전에는 `docs/development-environment.md`에 기본 프로파일을 기록한다.

## 프로파일 비교

| 항목 | `token-aware` | `full` |
|---|---|---|
| 시작 컨텍스트 | `HANDOFF.md`, CodeSight index, 관련 요구사항·정책 | 절약형 범위 + 관련 아키텍처·ADR·변경 이력 |
| 코드 탐색 | 대상 파일과 직접 의존성부터 점진적으로 확대 | 관련 모듈, 호출 경로, 유사 구현과 회귀 범위까지 탐색 |
| 사용자 질문 | 구현을 막는 질문만 묶어서 짧게 | 선택지, 장단점과 장기 영향까지 설명 |
| 설계 대안 | 유력안 1개, 중요한 대안이 있을 때만 비교 | 의미 있는 대안 2개 이상과 trade-off 비교 |
| 계획 | 짧은 실행 체크리스트 | 단계, 위험, 검증·rollback을 상세히 기록 |
| 구현 | 최소 diff와 기존 패턴 재사용 | 동일한 최소 구현 원칙 + 경계·확장 영향 심층 검토 |
| 테스트 | 변경 관련 필수 테스트와 위험 기반 검사 | 필수 테스트 + 넓은 회귀·통합·비기능 검사 |
| 문서 | 요구사항·결정·사용법의 필수 변경만 | 필수 문서 + 근거, 대안, 운영·마이그레이션 설명 |
| 사용자 보고 | 결과, 검증, 남은 위험 중심 | 결과 + 근거, 대안, 상세 검증과 후속 제안 |
| Handoff | 다음 세션이 재개할 최소 충분 컨텍스트 | 결정 과정과 조사 범위를 더 상세히 포함 |
| 프롬프트 템플릿 | 현재 작업을 막는 필수 필드만 질문 | 대안·장기 영향 필드까지 검토 |
| Eval | 관련 필수 regression subset 우선 | 관련 전체 regression·추가 trial·capability Eval |
| 유지보수·도입 | 역할·변경 계층과 관련 release만 확인 | upstream/downstream 호환성·migration 전체 검토 |

## 항상 유지하는 필수 게이트

다음은 토큰 절약 대상이 아니다.

- prompt injection과 외부 콘텐츠의 불신 처리
- secret scanner, SAST와 승인된 공급망 검사
- MCP server 승인 manifest, 미승인 호출 차단과 필수 권한·공급망 심사
- 설치된 package version 변경의 사용자 승인과 호환성·migration·회귀 검증
- 파일·DB 삭제, 배포, 권한 변경과 외부 전송의 사용자 승인
- 변경 파일의 포맷, 린트, 타입 검사와 관련 테스트
- 보안·개인정보·접근성 요구사항에 직접 관련된 검증
- 실제 Git diff 확인과 사용자 변경 보존
- 요구사항, 중요한 결정과 `HANDOFF.md` 갱신

전체 테스트가 매우 크다면 `token-aware`는 변경 관련 subset을 먼저 실행할 수 있다. 하지만
병합 필수 CI를 생략하거나 실행하지 않은 검사를 통과로 기록해서는 안 된다.

## `token-aware` 실행법

1. `HANDOFF.md`와 CodeSight index를 먼저 읽는다.
2. 관련 요구사항·보안 정책·대상 파일만 확인한다.
3. 검색으로 영향 경계를 찾고 필요한 파일만 추가로 읽는다.
4. 구현을 바꾸는 미확정 사항만 질문한다.
5. 기존 패턴을 재사용해 가장 작은 안전한 diff를 만든다.
6. 빠른 검사부터 실행하고 실패나 위험이 있을 때 범위를 확대한다.
7. 결과, 검증과 다음 시작점을 간결하게 handoff에 남긴다.

토큰을 줄이기 위해 원본 코드 확인 없이 CodeSight 요약만 믿거나 테스트를 추측하지 않는다.

## `full` 실행법

1. 절약형 시작 컨텍스트에 관련 ADR, 아키텍처와 변경 이력을 추가한다.
2. 직접 의존성뿐 아니라 소비자, 경계 조건과 기존 유사 구현을 확인한다.
3. 의미 있는 대안과 장단기 trade-off를 비교한다.
4. 기능, 회귀, 보안, 접근성, 성능과 운영 영향을 위험에 비례해 검증한다.
5. 중요한 근거와 기각한 대안을 ADR 또는 설계 문서에 남긴다.

`full`에서도 전체 저장소를 기계적으로 읽거나 같은 내용을 여러 문서에 복제하지 않는다.
CodeSight, 검색과 기존 문서로 탐색 범위를 먼저 좁힌다.

## 자동 전환과 중단 조건

`token-aware` 수행 중 다음 상황에서는 범위 확대가 필요함을 사용자에게 알린다.

- 인증, 결제, 개인정보, 암호화 또는 운영 데이터 변경
- 데이터 migration, 호환성 파괴, 복잡한 동시성 문제
- 문서와 실제 코드가 충돌하거나 영향 범위를 확인할 수 없음
- 반복되는 테스트 실패와 원인 불명 회귀
- 공급망·라이선스·텔레메트리 심사가 필요한 새 의존성

확대가 승인되지 않거나 토큰이 부족하면 안전하게 완료 가능한 범위까지만 수행하고 미검증
항목을 명시한다. 필수 게이트를 통과하지 않은 작업은 완료로 선언하지 않는다.

## 요구사항 변경 시 동기화

이 문서는 고정된 두 개의 템플릿이 아니라 현재 요구사항에서 파생되는 실행 정책이다. 새로운
요구사항이 들어오거나 기존 요구사항이 바뀔 때 다음 표를 기준으로 두 프로파일을 함께 검토한다.

| 검토 대상 | `token-aware` 갱신 질문 | `full` 갱신 질문 |
|---|---|---|
| 시작 컨텍스트 | 최소로 반드시 읽을 새 문서는 무엇인가? | 추가로 조사할 설계·이력은 무엇인가? |
| 사용자 질문 | 구현을 막는 필수 질문은 무엇인가? | 대안·장기 영향을 확인할 질문은 무엇인가? |
| 구현 | 가장 작은 안전한 적용 범위는 무엇인가? | 소비자·경계·마이그레이션 영향은 무엇인가? |
| 검증 | 생략할 수 없는 관련 검사는 무엇인가? | 확대할 회귀·통합·비기능 검사는 무엇인가? |
| 산출물 | 반드시 갱신할 문서와 handoff 항목은 무엇인가? | 추가할 ADR·근거·운영 문서는 무엇인가? |

변경 절차:

1. 요구사항 ID와 변경 유형을 확인한다.
2. 공통 필수 조건인지 프로파일별 실행 차이가 있는지 분류한다.
3. 이 문서와 `.ai/workflows/token-budget.md`의 관련 항목을 갱신한다.
4. 하네스, SDLC, 프로젝트 환경, 스킬과 hook·CI에 파급되는 내용을 반영한다.
5. 두 프로파일에서 완료 조건이 동일한 보안·품질 수준을 유지하는지 검증한다.
6. `HANDOFF.md`에 변경 결과 또는 영향 없음의 이유를 기록한다.

단순 오탈자처럼 실행 방식에 영향이 없는 변경은 표를 억지로 늘리지 않는다. 대신 프로파일
영향이 없음을 확인했다는 사실을 handoff에 남긴다.

## 개발환경 질문에 적용

- `token-aware`: 자동 감지 결과를 요약하고 현재 구현을 막는 `TBD`만 질문한다.
- `full`: 전체 질문 카탈로그를 점검하고 기술 선택의 대안, 운영 영향과 버전 정책까지 검토한다.

두 방식 모두 최종 선택은 `docs/development-environment.md`와 ADR에 기록한다. 질문 방식만
달라지며 확정된 환경의 재현성과 보안 기준은 달라지지 않는다.

## 추적성

- 관련 요구사항: REQ-017, REQ-018, REQ-020, REQ-021, REQ-022, REQ-023
- 이번 요구사항 영향: `token-aware`는 템플릿의 필수 필드만, `full`은 대안·장기 영향까지
  작성한다. 승인·인수 조건은 공통 필수다.
- Eval과 clone-onboarding 영향: 실행 범위는 달라질 수 있지만 환경 검증과 공통 보안·정확성
  grader는 두 프로파일 모두 필수다.
- 역할 분리 영향: 두 프로파일 모두 upstream 유지보수와 downstream 도입을 먼저 구분하며,
  공급망·비밀 분리·명시적 upgrade 조건은 공통 필수다.
- Git hook 적용 영향: 두 프로파일 모두 개발환경과 실제 품질 명령의 승인 전에는 Husky를
  설치하지 않는다. `full`은 추가 hook 대안과 호환성을 더 넓게 검토할 수 있다.
- Supabase·Firebase와 Human-in-the-loop 영향: `token-aware`도 key 경계, RLS·Rules, 데이터,
  비용과 production 승인 질문을 생략하지 않는다. `full`은 대안·lock-in·규제·복구 분석을
  확장한다.
- 관련 문서: [프로젝트 개발환경 정의](project-environment-definition.md),
  [하네스 구성](harness.md), [SDLC](sdlc.md), [Handoff](handoff.md)
